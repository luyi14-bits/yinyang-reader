P1 — 网络边界安全

专家：James Kettle（PortSwigger 研究院院长 · HTTP 协议漏洞挖掘）

F-01 · CRITICAL
无 Content-Security-Policy — 完全缺失
📁 reader.html · index.html · sw.js — 全 HTML 无 CSP meta 标签，GitHub Pages 无 CSP 响应头
当前 HTML 中不存在任何 CSP 声明（既无 <meta http-equiv="Content-Security-Policy">，GitHub Pages 服务器也不下发此响应头）。 这意味着浏览器不会对 inline script / inline style / eval / 外部资源来源做任何限制。 如果任何 JSON 数据被投毒或供应链被污染，攻击者可通过 innerHTML + <script> 标签获得完整 JS 执行权限。
<!-- 建议在 <head> 最顶部添加 --> <meta http-equiv="Content-Security-Policy" content=" default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data:; connect-src 'self'; manifest-src 'self'; frame-ancestors 'none'; base-uri 'self'; form-action 'none'; ">
F-02 · HIGH
无 X-Frame-Options — 存在 Clickjacking 风险
📁 reader.html · index.html — 无 frame 防御响应头
页面可以被嵌入任意 <iframe>。攻击者可将阅读器嵌入恶意页面，叠加透明层诱导用户操作（如双击触发 Sneak Mode、点击故事链接等）。 虽然当前页面无敏感交互，但未来若加入评论、打赏、登录等功能，此漏洞将升级为高危。
<!-- 添加 HTTP 响应头（通过 GitHub Pages 无法直接配置，建议加入 CSP frame-ancestors） --> # CSP 方案（推荐 — 已在 F-01 中包含 frame-ancestors 'none'）： Content-Security-Policy: frame-ancestors 'none'; # 备选方案： X-Frame-Options: DENY
F-03 · HIGH
无 X-Content-Type-Options — MIME 嗅探风险
📁 全 HTML — 缺失 nosniff 头
浏览器可能对响应进行 MIME 类型猜测。如果 Service Worker 缓存的数据被篡改（如伪造 Content-Type），浏览器可能将 JSON 解释为可执行的 HTML/JS。
<meta http-equiv="X-Content-Type-Options" content="nosniff">
F-04 · MEDIUM
Google Fonts 外部依赖 — 无 integrity 校验，用户 IP 泄露
📁 reader.html:19-21
字体从 fonts.googleapis.com 和 fonts.gstatic.com 加载。 (1) <link> 标签无 integrity 属性，字体 CSS 被篡改将直接注入页面上下文。 (2) 所有用户访问时 IP 地址会泄露给 Google。 项目自身 SECURITY.md 已识别此风险但未修复。
# 推荐方案 1：字体自托管（最佳） 将 Noto Serif SC 和 Ma Shan Zheng 的 woff2 文件下载到 /fonts/ 目录 使用 @font-face 本地引用 # 推荐方案 2：添加 integrity（次选 — 但 Google Fonts 动态生成 CSS，不易固定 hash） 在 CSP 中限制 font-src 和 style-src 为仅允许特定域名
F-05 · LOW
无 HSTS (Strict-Transport-Security) 头 — 依赖 GitHub Pages
📁 GitHub Pages 部署 — 无自定义 HTTP 头配置能力
GitHub Pages 默认强制 HTTPS 并发送 HSTS，但无法自定义 max-age 或 includeSubDomains。 此为 GitHub Pages 平台限制，风险可控。

🔴 P1 结论：不通过。核心安全头完全缺失。若部署到自建服务器，必须配置。

P2 — 输入输出审查

专家：James Kettle（延续 Web 安全视角）

F-06 · HIGH
6 处 innerHTML 赋值 — DOM XSS 攻击面
📁 reader.html · L1131, L1191, L1225, L1315, L1351, L1545

项目中有 6 处 使用 innerHTML 将动态数据写入 DOM。虽然当前数据来源为同源的静态 JSON 文件（非用户输入），但若 JSON 文件被供应链攻击替换或 Service Worker 缓存投毒，攻击者可注入恶意 HTML/JS。

最危险的 4 处：

  • L1131: content.innerHTML = story.content.map(p => ...) — 故事正文直接拼接进 HTML
  • L1191: bar.innerHTML = html — 分类标签（来自 INDEX.json categories 数组）
  • L1225: list.innerHTML = allStories.map(s => ...) — 故事标题 + 分类名 + 字数
  • L1351: list.innerHTML = filtered.map(email => ...) — Sneak Mode 邮件数据
// ❌ 当前（不安全）： content.innerHTML = story.content.map(p => '<p>' + p + '</p>').join(''); // ✅ 修复方案 1 — 使用 textContent 逐元素构建： content.innerHTML = ''; story.content.forEach(p => { const el = document.createElement('p'); el.textContent = p; // textContent 自动转义 HTML content.appendChild(el); }); // ✅ 修复方案 2 — 使用 DOMPurify 库（如必须用 innerHTML）： content.innerHTML = DOMPurify.sanitize(html); // ✅ 修复方案 3 — 对动态字符串做 HTML 实体编码： function escapeHtml(str) { const div = document.createElement('div'); div.textContent = str; return div.innerHTML; }
F-07 · LOW
story.title 在 library 列表中使用 textContent（安全），但在 story-card 中使用 innerHTML
📁 reader.html · L1125-1128 vs L1225-1236
reader 视图中的标题使用 textContent（安全），但 library 列表使用 innerHTML（不安全）。 同一份数据在不同路径上安全性不一致。
F-08 · LOW
Sneak Mode search input 设置为 readonly 但允许点击解除
📁 reader.html · L932 — onfocus="this.removeAttribute('readonly')"
Outlook 模拟搜索框仅用于伪装，不发起真实请求。输入内容不会被处理，无安全风险。但 removeAttribute 为不必要的动态操作。
检查项结果
eval() / new Function() 调用✅ 未使用
setTimeout/setInterval 字符串参数✅ 未使用
document.write()✅ 未使用
URL 重定向（location.href/open）✅ 仅 SW 更新时 reload()，无外部 URL

🟡 P2 结论：警告。innerHTML 在当前威胁模型（纯静态/同源数据/非用户输入）下风险有限，但防御纵深不足。强烈建议替换为 textContent 方案。

P3 — 存储 & 日志安全

专家：Daniel Miessler

F-09 · MEDIUM
生产代码大量 console 输出 — 信息泄露
📁 reader.html · 16条 / sw.js · 3条 / build.mjs · 5条

reader.html 包含 16 条 console 语句，泄露以下信息：

  • 故事文件加载路径（L1050, L1064）
  • 用户阅读完成记录（L1263）
  • SW 注册状态和 scope（L1587, L1592, L1598, L1602）
  • PWA 安装行为（L1563, L1577）
  • 触摸模式恢复状态（L1420, L1428, L1528, L1533）

攻击者若通过 XSS 获得执行权限，可通过覆盖 console.log 钩子拦截这些信息，辅助进一步侦查。

// 推荐：条件编译或构建时移除 console // 方案 1 — 全局开关： const DEBUG = false; const log = DEBUG ? console.log.bind(console) : () => {}; const warn = DEBUG ? console.warn.bind(console) : () => {}; // 方案 2 — 构建脚本中 strip console（当前 build.mjs 未做此处理） // 在 build.mjs 中读取 reader.html 并 regex 移除 console.* 调用
F-10 · LOW
localStorage 无加密 — 可被同源任意脚本读写
📁 reader.html · 6 处 localStorage 读写
存储内容仅为 UI 偏好，无敏感数据。但如果未来加入阅读进度同步、书签、用户备注等功能，应注意 localStorage 明文存储的风险。

🟡 P3 结论：警告。当前数据不敏感，风险可控。建议构建时自动 strip console 日志。

📍 总体评价 & 优先级修复建议

威胁模型

阴阳先生手记为纯静态 PWA，部署于 GitHub Pages（HTTPS），无后端、无数据库、无用户认证、无用户输入。攻击面天然较小。 核心威胁来自：

  1. JSON 数据文件被供应链攻击替换
  2. Service Worker 缓存投毒
  3. Google Fonts CDN 劫持
  4. 通过 XSS → 窃取或篡改 localStorage 数据
